# RemoteCC — Authentication Design
**Date:** 2026-04-05
**Status:** Approved — ready for implementation planning

---

## Context

RemoteCC is deployed on a Mac Mini exposed to the internet. Trusted users (owner + friends) access it via browser/PWA. The Agent runs locally and connects to the Server over the network. Currently there is no authentication on any endpoint.

---

## Decisions

### Mechanism: Passkeys (WebAuthn) for browsers, API key for Agent

Passkeys (FIDO2/WebAuthn) are used for all browser and PWA access — Touch ID on Mac, Face ID on iPhone, or platform authenticator on other devices. Keys sync via iCloud Keychain so registering once covers all Apple devices under the same Apple ID.

The Agent is a machine client and cannot participate in a WebAuthn ceremony. It authenticates to the Server using a pre-shared API key sent as an `X-Api-Key` request header. The Server's auth filter accepts either a valid session cookie or a valid API key.

**Rejected alternatives:** HTTP Basic Auth (poor UX, WebSocket support awkward), shared passphrase/token (less secure, no phishing resistance), external auth proxy (Caddy WebAuthn module not production-grade).

### Implementation: `quarkus-security-webauthn` extension

Quarkus ships a first-party WebAuthn extension. One interface (`WebAuthnUserProvider`) is implemented — everything else (challenge generation, CBOR parsing, assertion verification, session cookie issuance) is handled by the extension.

### User scope: shared access, invite-only registration

All authenticated users share the same access level. New users register via a one-time invite link generated by an already-authenticated user. No open registration.

### TLS: Caddy reverse proxy (deployment prerequisite)

WebAuthn requires HTTPS. TLS is handled by Caddy in front of Quarkus. This is a deployment requirement, not implemented in Quarkus itself.

## Architecture

### Protected routes
| Path | Protection |
|------|-----------|
| `/api/**` | `@Authenticated` (session cookie or API key) |
| `/ws/**` | Session cookie checked in `@OnOpen` |
| `/app/**` | `@Authenticated` → 302 to `/auth/login` if unauthenticated |

### Unprotected routes
| Path | Reason |
|------|--------|
| `/q/health` | Standard Quarkus health check |
| `/q/webauthn/**` | WebAuthn ceremony endpoints (managed by extension) |
| `/auth/login` | Login page |
| `/auth/register` | Registration page (invite token validated here) |

### New components (Server only)

**`CredentialStore`** — implements `WebAuthnUserProvider`. Reads/writes `~/.remotecc/credentials.json` (path configurable via `remotecc.credentials-file`). File reads happen on every lookup; writes are atomic (temp file + rename). Reads and writes synchronized on the instance. Supports multiple credentials per username (e.g. iPhone + Mac both registered).

**`InviteService`** — `@ApplicationScoped`. Maintains a `ConcurrentHashMap<String, Instant>` of UUID token → expiry. Tokens expire after 24 hours. Consumed on first successful use. Lost on server restart (acceptable — just generate a new invite).

**`AuthResource`** — JAX-RS resource at `/auth/*`:
- `POST /auth/invite` — requires `@Authenticated`; calls `InviteService.generate()`; returns `{"url": "https://.../auth/register?token=<uuid>"}`
- `GET /auth/register?token=<uuid>` — validates token via `InviteService`; serves registration HTML on valid token; 403 on invalid/expired

**`ApiKeyAuthMechanism`** — implements Quarkus `HttpAuthenticationMechanism`. Checks for an `X-Api-Key` header matching `remotecc.agent.api-key`; if present and valid, produces a security identity so `@Authenticated` passes normally. Runs alongside the WebAuthn cookie mechanism — Quarkus tries each registered mechanism in turn. Returns 401 JSON if the header is present but wrong.

**Login page** (`/auth/login`) and **register page** (`/auth/register`) — minimal HTML/JS using the WebAuthn browser API (`navigator.credentials.create` / `navigator.credentials.get`) against the Quarkus WebAuthn ceremony endpoints.

### Agent changes

`ServerClient` gains a `@ClientHeaderParam` annotation (or `ClientRequestFilter`) that injects `X-Api-Key: {value}` on every outbound request, reading from `remotecc.agent.api-key`.

## Credential Storage Format

File: `~/.remotecc/credentials.json` (created on first registration)

```json
[
  {
    "username": "mark",
    "credentialId": "<base64url>",
    "publicKeyCose": "<base64url>",
    "signCount": 42,
    "registeredAt": "2026-04-05T10:00:00Z"
  }
]
```

Multiple entries per username are allowed (multiple devices). The file is owned by the process user; no world-readable permissions.

---

## Bootstrap: First User

When `credentials.json` is empty (no credentials registered yet), `/auth/register` works **without an invite token** — the registration page is served unconditionally. Once any credential is stored, all subsequent registrations require a valid invite token. This is checked by `CredentialStore.isEmpty()` at request time.

## Invite Flow (subsequent users)

1. Owner authenticates via passkey → session cookie set
2. Owner calls `POST /auth/invite` → receives `{"url": "https://.../auth/register?token=<uuid>"}`
3. Owner sends URL via iMessage/Signal
4. Friend opens URL → `InviteService` validates token → registration page rendered
5. Friend enters display name → WebAuthn registration ceremony runs → credential stored in `credentials.json` → token consumed
6. Friend redirected to `/app/` with active session cookie

Invalid or expired token → 403 page: "This invite has expired — ask for a new one."

---

## Session Lifecycle

After successful WebAuthn login, the Quarkus extension issues a signed HttpOnly cookie. The signing key is generated at startup — server restarts invalidate all browser sessions (users re-authenticate with Touch ID). Credentials themselves persist across restarts in the JSON file.

The WebSocket upgrade is an HTTP request; browsers automatically include cookies, so `/ws/{id}` receives the session cookie. `@OnOpen` validates the security identity is present; unauthenticated connections are closed immediately before any data flows.

## Error Handling

| Scenario | Response |
|----------|----------|
| Unauthenticated REST request | 401 `{"error":"Unauthorized"}` |
| Wrong API key | 401 `{"error":"Unauthorized"}` |
| Unauthenticated `/app/**` page | 302 → `/auth/login` |
| Unauthenticated WebSocket upgrade | Connection closed in `@OnOpen` |
| Invalid/expired invite token | 403 page with human-readable message |
| WebAuthn ceremony failure | 400, handled by Quarkus extension |

Auth failures logged at WARN with remote IP. No stack traces in responses.

## Configuration Properties

```properties
remotecc.agent.api-key=<secret>          # Server: expected key in X-Api-Key; Agent: key to send
remotecc.credentials-file=~/.remotecc/credentials.json
```

Both Server and Agent read `remotecc.agent.api-key` from the same config namespace (same binary, different mode). Configure the same value on both sides.

## Testing

| Test | Type | What it covers |
|------|------|----------------|
| `ApiKeyFilterTest` | `@QuarkusTest` | Unauthenticated → 401; valid key → passes; wrong key → 401 |
| `InviteServiceTest` | Unit | Token generation, expiry, consumption, double-use rejection |
| `CredentialStoreTest` | Unit | Read/write round-trip to temp file; concurrent access; missing file initialises empty |
| `AuthResourceTest` | `@QuarkusTest` | Unauthenticated `POST /auth/invite` → 401; invalid token `GET /auth/register` → 403 |

WebAuthn ceremony flows are not unit-tested (crypto-heavy, browser-driven) — covered by manual smoke test against a real browser after deployment.

---

## Out of Scope

- Credential revocation UI (delete a user's passkey)
- Rate limiting on auth endpoints
- Explicit logout endpoint (session expires with cookie)
- Windows / Linux terminal adapters
